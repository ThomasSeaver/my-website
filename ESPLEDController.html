Arduino LED Controller

Controller for an ESP8266 (cheap wifi chipped Arduino) that's hooked up to individually addressable RGB LEDs. (WS2812Bs)

Why do this?

Normal lighting is boring. Mood lighting, lighting that automatically changes colors, and the ability to control it from any networked device drastically changes the atmosphere of a room as well as how smart you feel your home is. Unfortunately, most custom RGB lighting solutions are either prohibitively expensive, or garbage. The ones that require internet are either sending your data through whichever nationstates they want, or just putting insecure backdoors in your network. Sometimes both.

What's the solution, I hear you ask. My friend it is the magic of Arduino and RGB lights. If you don't know what an Arduino is, I'd suggest you go to the links below, but a short explanation is that it is a tiny computer that has accessible pins which makes it easy to control sensors or other devices which a bigger computer would have trouble interfacing with or utilizing.

In my setup, we use ESP8266s, AKA NodeMCUs. Beyond the names and setting the appropriate system in your IDE, this doesn't matter much. I chose them because they're about 5 bucks a pop on Amazon, and have wifi enabled. This is important because it means that as long as I'm on the same network, I can ping that device with requests for lighting, and it can respond automatically.

You simply run 5v power to a strip of LEDs (I use WS2812Bs, aka Neopixel knock offs. It doesn't matter what kind you use as long as you mark it in the code) and then run the data line on the strip to pin 6 on the arduino, which you must also run 5v power to. Originally I was running 5v power to the arduino, then sapping it for the lights across the board. This I do not recommend, as it worked for a bit than killed a chip, twice. Turns out there's a diode between Vin and the 5v, with a .6v drop. This means even if you're just pulling 2 amps, you end up with 1.2 watts or more on that tiny diode, and that's gonna cook your board.

After power has been run, the arduino is on, and you've set your network and what not correctly in the code, it should be connected. Either use the ip you set, or find it yourself, then begin pinging it with the various url encoded parameters I note in documentaiton to use it. This may not seem to be the best way to control it, but it's the simplest way I've found to give info to the Arduino through HTTP, and allows for other systems to control it through http requests.

That was important to me because I setup a Python server that could run on my computer automatically with Node.js, and be accessible from any device. It was a much nicer and easier way to present an HTML page than directly from the arduino while still being able to control it however I wanted. It also presents the option for further development to control multiple strips at once.

This was not the easiest process, but altogether for 30ish bucks you can have your own remotely controllable rgb led strip without having to send your data outside your network, and I think that's pretty neat. I'm pretty happy with it and looking forward to trying more Arduino stuff in the future.

Right now I just have the serial control program I built up, as it's the one I've worked on most recently and I ended up scrapping/redoing a lot of the code, but since it is controlled similarly, it should be trivial to go about updating it with better wifi code in the near future, when I get to test it back in my own network space again.
